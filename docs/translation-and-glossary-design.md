# BookReader: 全书翻译 + 名词解释 设计文档

> 自用工具先行，做好了再考虑是否开放。聚焦"每次打开书都有价值"的基础能力。
>
> **架构定位**：本文档是 `book-as-indexed-knowledge-base.md`（宪法）的**执行层**。翻译、术语、原话溯源在宪法中都是 Index 之上的不同 Query。读本文档前建议先读一遍宪法，理解 Parser / Index / Query / Application 的四层结构。

## 1. 目标与边界

### 1.1 目标

**给自己做一个读书时"不用切屏查东西"的工具。**

- 读外文书时，翻译质量够好，不需要自己再去 DeepL/ChatGPT
- 读任何书时，碰到专业术语/人名/概念，直接看懂
- 读的书越多，积累的个人知识资产越厚

### 1.2 为什么是这两个功能

```
相比深度学习 Agent（强制回忆、间隔复习等）:
├── 价值每次打开就有, 不需要长期验证
├── 失败成本低（最差就是翻译一般, 不影响阅读）
├── 不依赖"接受困难"这种存疑假设
└── 建立在现有翻译功能上, 是升级不是重建

相比普通翻译 App:
├── "全书" = 术语一致性 + 语境连贯 + 风格统一
├── 名词解释 = 不只是翻译, 是理解背景
└── 个人术语表 = 个人知识资产, 读越多越有价值
```

### 1.3 不做什么（避免范围蔓延）

- ❌ 强制回忆、间隔复习等学习 Agent —— 放到后面
- ❌ 社交分享、社区功能 —— 自用先不考虑
- ❌ GEO、商业化 —— 做好了用着爽再谈
- ❌ 完美支持所有语言对 —— 先做自己常读的（比如 EN→ZH）
- ❌ 复杂的权限/账号系统 —— 自己用

## 2. 核心功能

### 2.1 全书翻译

**关键差异：不只是"把每句话翻一遍"**

```
普通翻译:
  逐句/逐段独立翻译
  问题: 同一术语不同章节译法不同, 风格飘

全书翻译:
  Step 1: 扫描全书，构建"术语表 + 风格档案"
  Step 2: 基于术语表和风格，按章节翻译
  Step 3: 存储原文 + 翻译 + 术语映射

优势:
├── 术语一致（"hacker"全书统一为"黑客"）
├── 核心意象保留（"the green light"→"那盏绿灯"不是"绿色的灯"）
├── 风格统一（学术/口语/文学不混用）
└── 可二次校正（用户改一处，全书同步）
```

### 2.2 名词解释

**层次化解释，不是单一长文**

```
识别对象:
├── 专有名词（人名、地名、事件）
├── 专业术语（学科概念）
├── 文化典故（俚语、历史引用）
└── 作者特定用法（自创术语）

解释深度（可展开）:
├── 一行（悬停/点击时显示）: 核心定义
├── 一段（展开）: 背景 + 语境
└── 一页（深入）: 历史、争议、延伸阅读

识别方式:
├── 翻译时 LLM 同步识别（一次调用双用途）
├── 用户手动标记（划线 → 标为"待解释"）
└── 未来: 基于用户知识水平自动识别
```

### 2.3 个人术语表（Personal Glossary）

**这是最有长期价值的部分，做到位后是差异化护城河**

```
累积逻辑:
  读书 A 碰到"涌现" → 记录到术语表
    ├── 定义
    ├── 出现上下文（书 A 第 3 章原文片段）
    ├── 我的理解（手动加）
    └── 关联概念
  读书 B 碰到"涌现" → 扩充同一条目
    ├── 新增书 B 的上下文
    ├── 两本书定义的差异
    └── 我的对比理解

结果: 术语表 = 个人的知识图谱
     读越多, 越有价值
     别的阅读 App 做不到这个
```

> **宪法对应**：术语表在宪法里就是 Index 的 `concepts` + `occurrences` 两张表。"跨书聚合"就是 `workspaceSymbols` + `concept_merges`。本小节的"累积逻辑"是 Index Layer 的自然产物，不需要单独再设计数据结构。

### 2.4 原话溯源（Author's Own Words）

**这是术语功能的"免费升级"，也是 MVP 必做项**

```
痛点场景:
  读到第 8 章 "作者利用 ATR 计算仓位..."
  → 忘了 ATR 到底是什么
  → 传统做法: 翻回去找, 痛苦

有原话溯源:
  点击 "ATR" → 浮层立刻显示:
  ├── 作者首次定义 @ 第 2 章 p.28 的原文摘录
  ├── 作者后续补充过 3 次 (可展开)
  └── 作者使用过 9 次 (可展开)
  → 零 AI 生成, 零幻觉, 纯原文检索
```

**和术语解释的关键区别**：

| 维度 | 术语解释（AI 生成） | 原话溯源（索引检索） |
|------|-------------------|-------------------|
| 内容来源 | LLM 综合知识 | 作者原文摘录 |
| 幻觉风险 | 中 | 无 |
| 权威度 | 辅助 | 最高 |
| 运行成本 | 每次调用 LLM | 扫描一次，查询零成本 |
| 适合场景 | 作者没讲清 / 要额外背景 | 作者讲过但读者忘了 |

**产品上二选一的清晰二元对立**：

```
用户选中术语 → 浮层两个按钮:
  [📖 作者怎么说]   [💬 讲讲]
       ↑                 ↑
   原文检索          AI 生成
   零幻觉            有幻觉风险
```

**分类展示（关键差异化）**：

扫描时对每次出现标注类型：
- 📌 **定义性出现**（definition）：作者在下定义，通常 1-2 处
- 💡 **补充/深化**（refinement）：作者后续补充、对比、修正
- 🎯 **应用/使用**（usage）：作者在调用这个已定义概念

用户需求不同：
- "ATR 是什么？" → 📌（一眼看够）
- "完整的讲解？" → 📌 + 💡
- "用在哪些场景？" → 🎯
- "所有出现？" → 全部

如果不分类直接展示 20 条，反而难用。

**宪法对应**：
- 悬停预览 = `hover(conceptId)`
- 首次定义跳转 = `goToDefinition(conceptId)`
- 所有出现列表 = `findReferences(conceptId, {types})`
- 跨书同概念的出现 = `crossBookOccurrences(conceptId)`

## 3. 关键设计决策（附我的推荐）

### 3.1 翻译时机

| 选项 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| 打开即全书翻译 | 读起来顺畅 | 首次等待长 | ❌ |
| 按章节懒加载 | 打开快 | 切章可能等待 | ⚠️ 次选 |
| **后台渐进翻译** | 打开快 + 几乎无等待 | 实现复杂度中 | ✅ 推荐 |

**实现思路**：打开书时立即翻译当前章节（3-5 秒），后台异步翻译其他章节。用户向后翻时大概率已翻译完成。

### 3.2 翻译 + 解释是一次调用还是两次

| 选项 | 优点 | 缺点 |
|------|------|------|
| **一次调用（推荐 MVP）** | 成本低，术语和翻译一致 | 解释可能泛泛 |
| 两次调用（后期） | 解释可深入、可个性化 | 成本翻倍 |

**MVP 先做一次调用**，后期如果觉得解释不够深，再升级为"快速一次 + 按需二次深入"。

### 3.3 显示方式

```
推荐: 段落下方显示翻译（对应 2.2 的 B 方案）

理由:
├── 阅读流畅（不用左右扫视）
├── 手机友好（屏幕窄）
├── 容易对比（上下相邻）
└── 可一键隐藏/切换

术语在文中用不同颜色/下划线标记，
悬停/点击显示简短解释，再点展开详细。
```

### 3.4 术语解释的默认深度

```
默认: 一行（核心定义）
理由: 不打断阅读流, 想了解更多再展开

展开策略: 逐级加深
  一行 → 一段 → 详细

不要: 默认就展示长篇大论 - 会变成"被迫学习"而不是"辅助理解"
```

## 4. 差异化方向

这些是"做好 vs 做差"的分水岭，不是 MVP 必须，但决定长期价值：

### 4.1 术语一致性（MVP 必须做好）

**反面教材**：同一本书，"consciousness" 第一章译"意识"，第五章译"觉察"

**实现**：翻译前先构建术语表（LLM 扫一遍全书），翻译时强制使用术语表，不允许 LLM 临时换译法。

### 4.2 用户可校正（P1）

**场景**：你读 Peter Drucker，AI 把 "effectiveness" 译成"有效性"，你觉得"成效"更贴切。

**实现**：
- 对某段翻译"编辑"
- 识别到是术语修正后，弹出"是否全书替换？"
- 用户确认后，全书该术语统一更新

**价值**：从"全自动"变"半自动"——你是主人，不是接受者。

### 4.3 个人术语表的跨书聚合（P1-P2）

**场景**：你在 3 本不同的书里读到"涌现"，每本定义略有差异。

**实现**：
- 术语表按"概念"聚合，不是按"书"分开
- 一个概念下挂着它出现的所有上下文片段
- 可以对比不同作者的定义

**价值**：读到第 N 本书时，回看发现自己对这个概念的理解是如何演进的。

### 4.4 知识水平自适应（P2+）

**场景**：你是程序员，读技术书时不需要解释"递归"；读经济学书时需要解释"边际效用"。

**实现**：
- 术语解释时判断用户对该领域的熟悉度
- 基于：用户读过哪些书、哪些术语已经在个人术语表中、是否划线过相关内容
- 判断后决定：是否需要解释、解释深度

**价值**：不被"已经知道的概念"打扰，节省注意力。

## 5. 技术架构

### 5.1 数据流（四层架构）

对齐宪法的分层模型：

```
┌─────────────────────────────────────────────────┐
│ Application Layer                               │
│  阅读器 UI / 翻译显示 / 术语浮层 / 溯源面板      │
├─────────────────────────────────────────────────┤
│ Query Layer                                     │
│  hover / goToDef / findRefs / searchSymbols     │
├─────────────────────────────────────────────────┤
│ Index Layer                                     │
│  concepts / occurrences / translations / ...    │
├─────────────────────────────────────────────────┤
│ Parser & Extractor                              │
│  EPUB 解析 + LLM 术语扫描 + 出现类型标注         │
├─────────────────────────────────────────────────┤
│ Raw Content                                     │
│  EPUB 文件                                       │
└─────────────────────────────────────────────────┘
```

具体流程：

```
EPUB 文件
  │
  ▼
Parser (零 LLM 成本)
  ├── 解析为 chapters → sections → paragraphs
  ├── 为每个 paragraph 生成稳定 ID (content-hash + 位置)
  └── 输出: AST + 段落锚点
  │
  ▼
Extractor Phase 1: 术语扫描 (一次性, LLM)
  ├── 输入: 章节摘要 + 目录
  ├── LLM 识别候选术语
  └── 输出: concepts 初稿 (term, category, initial_definition)
  │
  ▼
Extractor Phase 2: 按章处理 (可并行, LLM)
  ├── 输入: 章节原文 + 术语表 + 风格档案
  ├── LLM 产出:
  │    ├── 译文 + term_spans (挂到 translations 表)
  │    ├── occurrences: 每次出现的 type (定义/补充/使用)
  │    ├── context_before / context_after
  │    └── confidence 标注
  └── 输出: 结构化 JSON → Index Layer
  │
  ▼
Index Layer (持久化)
  ├── concepts / occurrences / relations
  ├── translations (挂在 paragraph 上)
  └── user_revisions (用户修订持久化)
  │
  ▼
Query Layer (零 LLM 成本, 全检索)
  ├── hover(conceptId)
  ├── goToDefinition(conceptId)
  ├── findReferences(conceptId, {types})
  ├── searchSymbols(query)
  └── workspaceSymbols(query) [跨书]
  │
  ▼
Application (UI 渲染)
```

**关键原则**（源自宪法）：
- Extractor 产出的必须是**结构化 JSON，不是自由文本**
- 所有 Index 数据带 `confidence`
- 所有应用层功能**只能通过 Query Layer 访问 Index**

### 5.2 技术栈

基于你现有的 BookReader 架构（`epub-tts-backend` + `epub-tts-frontend`）：

```
后端扩展:
├── 新增服务: TranslationService
│   ├── 术语扫描
│   ├── 章节翻译
│   └── 校正应用
│
├── LLM 调用层
│   ├── 推荐: Claude 3.5 Sonnet / Sonnet 4（翻译质量好）
│   ├── 成本敏感场景: 术语扫描可用 Haiku
│   └── 保留接口抽象（方便后换模型）
│
└── 存储 (完整 schema 见宪法 §6, 存储选型见 §6.1)
    ├── 引擎: SQLite + FTS5 + JSON1  (v0 / v1)
    │   └── per-book: ~/.bookreader/books/<book>.bookindex
    │   └── 全局:    ~/.bookreader/global.db  (书架 + 跨书映射)
    ├── 迁移路径: PostgreSQL + pgvector (v2+ 向量搜索时)
    │
    ├── books              书元信息
    ├── paragraphs         段落 (稳定 ID, 跳转锚点)
    ├── concepts           概念/术语 (带 aliases, confidence)
    ├── occurrences        每次出现 (带 type: definition/refinement/usage)
    ├── relations          概念关系图
    ├── translations       段落级翻译 (含 term_spans)
    ├── concept_merges     跨书聚合 (可逆)
    └── user_revisions     用户修订持久化 (审计)

前端扩展:
├── 阅读器模式切换: 原文/译文/双语
├── 术语标记和交互（悬停/点击）
├── 术语浮层 (含原话溯源 + AI 讲解 两入口)
├── 溯源面板 (分类列出所有 occurrence)
└── 术语索引页 (Outline 视图)
```

**Schema 要点**（不复述宪法，只强调这份 MVP 的特殊关注）：

- `paragraphs.id` **必须稳定**——用 content-hash + 位置签名生成，不依赖字节偏移。用户批注和跳转锚点全靠它。具体算法见 `paragraph-id-design.md`。
- `concepts.aliases` 原生支持——"ATR" / "Average True Range" 属于同一 concept。
- `occurrences.occurrence_type` 是原话溯源功能的关键字段，扫描时必须产出。
- `translations` 挂在 `paragraph_id` 上，不是章节级——这样修订译文时能精确定位。

### 5.3 LLM Prompt 设计要点

**术语扫描的 Prompt**（Phase 1）：
- 输入：章节摘要 + 目录
- 要求：识别"如果不理解会影响阅读"的专业术语、专有名词、作者自创概念
- 输出：结构化 JSON（term, aliases, category, initial_definition, confidence）
- 关键：限制数量（比如每章 5-15 个），避免把常识词也算进去

**章节处理的 Prompt**（Phase 2，翻译 + 出现标注合并）：
- 输入：章节原文 + 术语表 + 上下文（前一章摘要）
- 要求：
  - 译文严格使用术语表中的译法
  - 保持风格一致（学术/口语/文学，由 Phase 1 判断）
  - 在译文中标注术语位置（term_spans）
  - **对每次术语出现标注类型**：`definition` / `refinement` / `usage`
  - **提取 context_before / context_after**（前后 1 句），用于溯源预览
  - 每项产出带 `confidence` (0-1)
- 输出结构：
  ```json
  {
    "translated_text": "...",
    "term_spans": [{ "concept_id", "start", "end" }],
    "occurrences": [
      {
        "concept_id": "atr",
        "paragraph_id": "...",
        "occurrence_type": "definition",
        "is_first_def": true,
        "context_before": "...",
        "context_after": "...",
        "extracted_note": "作者把 ATR 定义为 ...",
        "confidence": 0.92
      }
    ]
  }
  ```

**出现类型的判定规则**（告诉 LLM）：

```
definition:  作者在下定义
  信号词: "is defined as" / "我们把…称为" / "所谓…指的是"
  特征: 该段落主要目的就是解释这个概念

refinement:  作者补充或对比一个已定义概念
  信号词: "回到…" / "前面没提到的一点是…" / "要注意…的区别"
  特征: 概念已被定义, 作者在深化

usage:       作者在使用
  特征: 概念只是工具, 不是讨论对象
  "计算 ATR 为 1.5" / "根据前面的 ATR…"

原则: 拿不准就标 usage + 低 confidence, 让用户修订
```

**解释的 Prompt**（按需，场景 A "讲讲"）：
- 输入：术语 + 当前段落上下文 + `findReferences` 返回的作者原话 + 用户知识画像（如果有）
- 要求：
  - **先引用作者原话**，再做补充解释
  - 三档长度（一行/一段/详细）
  - 无法引用原文的判断标注为"推测"
  - 结合书中的使用场景，避免泛泛而谈
- 输出：结构化（short, medium, detailed, cited_occurrences）

## 6. 实施路径

对齐宪法的 v0 → v1 → v2 → v3 路径。原 M1-M4 被吸收、重排。

### v0: 索引地基（2-3 周）

```
目标: 跑通 Parser → Extractor → Index → 最基础 Query
      一本书可以 hover/跳定义/查全部出现

范围:
├── EPUB 解析为 paragraphs (稳定 ID)
├── LLM 两阶段扫描 (术语候选 + 章节处理)
├── Index 持久化: concepts + occurrences + paragraphs
├── Query Layer 最小实现:
│   ├── hover(conceptId)
│   ├── goToDefinition(conceptId)
│   └── findReferences(conceptId, {types})
└── 最简 UI:
    ├── 术语高亮
    ├── 悬停浮层: 显示首次定义 + 出现统计
    └── "所有出现"列表 (分类折叠)

不做: 翻译、跨书、Agent 讲解、用户修订

验证方式:
├── 对 3 本你常读的书 (含 "走进我的交易室") 跑扫描
├── 自己读一周, 看原话溯源是否真的好用

成功标准:
├── 碰到忘了的术语, 点一下就懂, 不再翻回去
└── "ATR 的所有出现" 分类清晰, 不是一堆无序列表
```

## 7. 成本估算

### 7.1 单本书成本（以 30 万字为例）

```
术语扫描（1-2 次调用，输入为章节摘要）:
  ≈ 20K input tokens
  成本: ~¥0.5 (Sonnet)

章节翻译（输入原文 + 术语表，输出译文 + 标记）:
  ≈ 30万字 × 1.5 (tokenization) = 45万 tokens 输入
  ≈ 45万 tokens 输出
  成本: ~¥10-15 (Sonnet)

术语解释（每个术语一次，假设 100 个术语）:
  ≈ 100 × 2K tokens = 20万 tokens
  成本: ~¥5 (Sonnet) 或 ~¥1 (Haiku)

单本书总成本: ¥15-20
```

### 7.2 自用年度预算

```
假设一年读 20 本书:
  翻译 + 解释成本: ¥300-400

考虑复用（同一本书反复读不重算）和优化:
  实际: ¥200-300/年

这比订阅一个 AI 服务还便宜
```

### 7.3 降本优化方向

```
可以做但不必立刻做:
├── 术语扫描用 Haiku (省 80%)
├── 常见术语的解释全局缓存 (相同术语不同书共享)
├── 术语表预构建 (热门书提前处理)
└── 自托管开源模型 (前期不推荐, 质量损失大)

暂不考虑:
└── 本地 LLM 翻译 (质量差距明显, 电池消耗大)
```

## 8. 未决问题（需要边做边决定）

以下是开始做之后才能有答案的问题，记下来：

```
1. 术语扫描的粒度
   - 一章扫一次 vs 全书扫一次?
   - 一章太细, 会漏跨章概念
   - 全书太粗, 大书 context 超限
   - 解: 试试看, 可能需要两阶段

2. 术语表的"收录标准"
   - 多专业才算术语?
   - 人名都算 vs 只算作家提到的人?
   - 解: 先宽松, 观察自己实际用哪些, 调整

3. 翻译风格一致性的验证方法
   - 怎么判断全书风格是否统一?
   - 能否量化?
   - 解: 早期靠人工抽检, 稳定后可以做自动评估

4. 跨书术语聚合的判定
   - "涌现" 和 "Emergence" 在不同书里, 怎么判断是同一概念?
   - 简单匹配 vs 语义匹配?
   - 解: MVP 用严格字面匹配, 后期语义扩展

5. LLM 幻觉的防范
   - LLM 编造术语解释怎么办?
   - 解: 要求 LLM 引用原文出处, 无法引用的标记"需人工确认"

6. EPUB 格式的兼容性
   - 不同出版的 EPUB 结构差异大
   - 解: 先用自己常读的几本书的格式, 逐步扩展
```

## 9. 决策日志（会持续更新）

| 日期 | 决策 | 理由 |
|------|------|------|
| 2026-04 | 自用先行，不做用户体系 | 先验证自己用不用，别的是后话 |
| 2026-04 | 先做 EN→ZH | 自己主要需求 |
| 2026-04 | 基于现有 epub-tts-* 架构 | 不重建 |
| 2026-04 | LLM 选 Claude Sonnet | 翻译质量好，成本可接受 |
| 2026-04 | MVP 一次调用双用途 | 省钱，后期再拆 |
| 2026-04 | 引入"架构宪法"分层 | 翻译/术语/溯源统一为 Index 上的 Query，避免烟囱式设计 |
| 2026-04 | MVP v0 优先做"原话溯源"而非翻译 | 零幻觉、零运行成本、高频使用；翻译是 v1 |
| 2026-04 | 扫描时强制标注 occurrence_type | definition/refinement/usage，是溯源分类展示的关键 |
| 2026-04 | paragraph_id 必须稳定 | 跳转锚点 + 批注锚点都靠它；用 content-hash + 位置签名 |
| 2026-04 | 所有 LLM 产出带 confidence | 低置信触发用户确认，用于 UI 可信度标注 |
| 2026-04 | 跨书聚合只"推荐"不"自动执行" | 误合并侵蚀信任，必须用户确认且可逆 |
| 2026-04 | 存储选 SQLite，不用 MongoDB | 数据本质是强关系（JOIN 多 + 全文搜索 + 图遍历），JSON 字段少量用 JSONB 足够；pgvector 未来迁移无缝 |
| 2026-04 | per-book `.bookindex` 文件组织 | 索引是用户资产；跨书查询用 `ATTACH DATABASE`；契合宪法"索引是用户资产"原则 |
| 2026-04 | 不引入 Neo4j | 图结构浅（1-3 跳），SQL 递归 CTE 足够；数据割裂带来的一致性/运维负担大于收益；未来用 Apache AGE 平滑升级 |
