## 2. Prompt 设计

完整 prompt 见 `experiments/parser-prototype/prompt_phase1.md`。

核心结构：

```
【角色】学术术语抽取助手 (Book Language Server)
【任务】从章节文本识别核心概念
【原则】
  - 必须抽取: 核心术语 / 被引用的人名 / 引用作品
  - 不要抽取: 常识词 / 例子人物 / 普通描述
【分类】term / term_custom / person / work / theory
【输出】严格 JSON (含 reasoning 和 confidence)
【数量】宁少勿多, 每 10 段 ≤ 5 个
```

### 关键设计决定

1. **要求 `reasoning` 字段** — 一句话说明抽取理由，大幅降低 debug 成本
2. **`confidence` 是核心性 (not existence)** — 明确定义为"这是本章核心概念"的把握
3. **`first_occurrence_paragraph` 用 `P##` 格式** — 段落 ID 用前缀标记送进去，让 LLM 能精确指认
4. **数量上限软约束** — "每 10 段 ≤ 5 个"作为经验值，不是硬 cap
5. **category 五元体系** — term / term_custom / person / work / theory（有自创术语单独标）

## 5. 对 Prompt 的改进建议

### 5.1 发现 1：章节标题被误当段落

```
[P00|c7505f626aae-000] 第一章 有条件养育
```

章节标题本身就是第 0 段。应在 prompt 中明确："首段若与章节标题相同，通常是标题重复，不作为 concept 首次定义位置"。

### 5.2 发现 2：缺少全书语境

Prompt 只给了章节文本，LLM 无法判断"这个概念在后续章节会不会再出现"。

```
实际发生: 撤回爱 confidence = 0.65
理想结果: 0.85+ (因为下一章就是主题)
```

改进方向：
- 在 prompt 输入附带"目录概览"（全书章节标题列表）
- 让 LLM 根据后续章节标题推断重要性

### 5.3 发现 3：Concept 别名策略不明确

抽取了 `"情感虐待"` 的 alias 是 `["冷暴力", "emotional abuse"]`。但：
- "冷暴力" 是原文里用的（"情感虐待（冷暴力）"）
- "emotional abuse" 是我作为 LLM 自己翻的

Prompt 应明确：**aliases 只收录原文出现过的变体，翻译留给 Phase 2**。

### 5.4 发现 4：first_occurrence_paragraph 的精度

- "爱丽丝·米勒" first_occurrence = P02 ✓
- "戴安娜·鲍姆林德" first_occurrence = P21 ✓ （都在小节 2，pidx 计数从 0 开始）

**但**：Prompt 输入的是三小节拼接的文本，P 编号重复。

```
[P02] 小节 1 第 2 段
[P02] 小节 2 第 2 段   ← 冲突
```

改进方向：跨小节需要统一编号系统，比如 `S1P02` 或直接 `g023`（全局计数）。

### 5.5 发现 5：reasoning 字段价值高，不应删除

原计划上线前删掉 reasoning 省 token，但实际使用中它对审计、debug、用户信任都有价值。

**新判断**：保留 reasoning，**值得那几百 tokens**。

## 6. Prompt v2 迭代方向

基于上述，下一版 prompt 需要：

```diff
+ 输入附加: 全书目录 (提供后续章节语境)
+ 明确: aliases 只收原文出现的变体
+ 统一: 段落编号跨小节全局化 (G001, G002, ...)
~ 保留: reasoning 字段
~ 保留: confidence 定义
```

其他待验证：
- Prompt 是否在英文书上也能稳定（需要跑 Elder 一章验证）
- 数量上限是否需要更严（本章 13 个偏多？）
- 能不能只给章节"第一段 + 中间一段 + 最后一段"简化输入（降成本）

## 7. Phase 1 → Phase 2 衔接准备

Phase 1 产出的 13 个 concepts 将作为 Phase 2 的输入。Phase 2 任务：

```
对整章每个段落:
  1. 翻译 (如果需要)
  2. 扫描本段有没有提到这 13 个 concepts 之一 (通过 term 或 aliases 匹配)
  3. 对每次出现, 标注:
     - occurrence_type: definition / refinement / usage
     - context_before / context_after (前后一句)
     - confidence
  4. 产出 translations + occurrences
```

这才是 Index 的血肉。本次 Phase 1 验证了"术语识别"环节可行，Phase 2 原型下一步做。

## 8. 运行成本估算

本次实际运行（我作为 Claude Opus 4.6）：

```
输入:  ~ 12K tokens (prompt + 章节文本)
输出:  ~ 2K tokens (JSON with reasoning)
成本估算 (Sonnet 价位):
  输入 12K × $3/1M = $0.036
  输出 2K × $15/1M = $0.030
  本次一章扫描 ≈ $0.066 ≈ ¥0.5
```

推算到全书：《无条件养育》10 章，Phase 1 术语扫描 ≈ ¥5。
加上 Phase 2 翻译+标注 ≈ ¥15-20。

符合原设计文档 §7 的成本估算。
